# Loose (Docker-auto-named) container grouping

## Problem

A homelab host accumulates orphan standalone containers that were started without an
explicit `--name`, so Docker assigns each a random `adjective_surname` name
(`adoring_saha`, `cool_ride`, ...). dockbrr models every standalone container as a
one-service project, so each becomes its own sidebar entry and its own dashboard group.
The sidebar and table fill with throwaway noise, burying the projects the user actually
cares about.

The signal that separates noise from signal is the **name**: a Docker-assigned name means
the user never named the container, so it is almost certainly disposable. An explicitly
named standalone container (e.g. `backrest`) is one the user cares about and must stay a
normal top-level entry, regardless of run state.

## Design

Classify each standalone container by whether its name was Docker-assigned, carry the
result as a `auto_named` flag on the project, and render auto-named ("loose") projects
inside a single collapsed-by-default `Loose (N)` group in both the sidebar and the table.
Nothing is hidden permanently: loose containers stay discovered, tracked, and one click
away.

### 1. Name classifier (Go)

New package `internal/discovery/dockername`:

- Embeds copies of moby's two word lists (adjectives, surnames) as
  `map[string]struct{}` sets. The upstream `namesgenerator` package is officially
  **frozen** (no new names will ever be added), so the copied lists never drift and need
  no maintenance. Header comment cites the source + frozen status; lists are MIT
  (moby/moby).
- `IsDockerAssigned(name string) bool`:
  1. Strip a single optional trailing digit (`focused_turing3` -> `focused_turing`): the
     collision suffix that `GetRandomName` appends.
  2. Split on the first `_` into `adjective`, `surname`.
  3. Return `adjective ∈ adjectives && surname ∈ surnames`.

  A false positive requires a user to name a container *exactly*
  `<docker-adjective>_<docker-surname>` (~25k combinations); treated as acceptable.
  `boring_wozniak` (never generated by Docker) still classifies as auto-named if seen,
  which is harmless.

### 2. Flag (store + discovery)

- Migration `0009_projects_auto_named.sql`:
  `ALTER TABLE projects ADD COLUMN auto_named INTEGER NOT NULL DEFAULT 0;`
- `store.Project` gains `AutoNamed bool`; `Upsert`/`Get`/list read+write the column
  (mirrors the existing `Unmanaged` bool wiring).
- Discovery sets, on reconcile:
  `AutoNamed = kind == "standalone" && dockername.IsDockerAssigned(name)`.
  **Gated to `standalone`**: a compose project is never auto-named, so a real
  multi-service project can never be swept into the Loose group, even if its directory
  name happened to match the pattern. Independent of run state: a running loose
  container is still loose.

### 3. API + types

- The project payload in `internal/httpapi` gains `auto_named bool`.
- `web/src/api/types.ts` `Project` gains `auto_named: boolean`.

### 4. Sidebar (`SidebarProjects`)

- Partition `projects` into `named` (`!auto_named`) and `loose` (`auto_named`).
- Render `named` exactly as today.
- When `loose.length > 0`, render one collapsible group at the bottom of the list:
  a `Loose (N)` header row (disclosure caret + count), **collapsed by default**; expanding
  reveals the loose project links rendered with the existing row markup.
- Collapse state is component-local (`useState`), default collapsed. Not persisted.

### 5. Table (`DashboardTable`)

- `joinRows` already yields per-project header + service rows and the table already
  supports collapsible project groups. Add a sibling **Loose group**: rows belonging to
  `auto_named` projects are pulled out of the normal flow and rendered under a single
  collapsible `Loose (N)` section, **collapsed by default**, after the normal projects.
- No new Filters control (`Show unnamed` toggle is explicitly *not* added).
- **Filter interaction:** the Loose group participates in the existing filters. When a
  search term or a non-`any` status filter is active, the Loose group **auto-expands** so
  matches are never hidden behind a collapsed header; with no active filter it stays
  collapsed. The `(N)` count reflects post-filter matches. A loose group with zero matches
  under an active filter is omitted (parity with the existing "drop empty project header"
  rule).

### Not doing

- No discovery-level ignore: loose containers remain tracked and scannable, so the change
  is fully reversible and a container that later gets an explicit name simply leaves the
  group.
- No per-service backend flag: the flag lives on the project. A standalone project's name
  *is* its container name, so project-level classification is exact for the only case that
  can be auto-named.
- No Filters-bar toggle: collapse-by-default is the entire "hide by default" mechanism, in
  both surfaces.

## Testing

- `dockername` unit tests: real auto-names match (`adoring_saha`, `focused_turing`);
  trailing-digit collision names match (`focused_turing3`); explicit names do not
  (`backrest`, `postgres`); compose-style names do not (`myproj_web_1`); non-pattern
  strings do not (`web_app` where either half is outside the lists).
- `store`/`discovery`: `AutoNamed` is set for a standalone container with a Docker name,
  false for an explicitly named standalone, false for compose services; round-trips
  through `Upsert`/`Get`.
- `SidebarProjects.test.tsx`: named projects render at top level; auto-named projects
  render only inside a `Loose (N)` group that is collapsed by default and expands on
  click.
- `DashboardTable.test.tsx`: auto-named project rows render inside the collapsed Loose
  group by default; the group auto-expands when a search/status filter is active; a
  filter that matches nothing loose omits the group.
