# dockbrr: Repository audit & Phase-8 plan seed

**Date:** 2026-07-04
**Baseline:** branch `master` @ c24242e (Phase 7 complete, whole-branch review merged); `go build ./...` + `go vet` clean; web suite 57/57 green.
**Reference design:** `docs/dev/specs/2026-06-30-dockbrr-design.md`

## Verdict

The core safety architecture holds everywhere audited: the detection path never touches Docker, compose verbs are whitelisted and exec'd as argv (no shell), a snapshot precedes every mutation, pull-before-up is enforced, and the per-project keyed mutex is real. The problems are in feature completeness and UI finish, not safety.

Findings are grouped: **A** = broken/inert functionality (bugs), **B** = missing vs design (scope gaps), **C** = UI polish (density, chrome, feedback). Each carries an implementation plan. A suggested execution order closes the document.

## A. Broken or inert functionality

### A1. Manual project add is inert, no services ever created

**Where:** `internal/httpapi/settings.go:169` (`handleCreateProject`), `internal/discovery/discovery.go:216` (reconciler manual-skip guard).

**Problem:** `handleCreateProject` validates the compose files and stores only the project row. The reconciler skips `source=manual` projects entirely, so nothing ever creates service rows for them. Result: no detection, no updates, an empty header on the dashboard. The feature is a dead end-to-end path, contradicting design §2 "Workload intake: auto-discover + manual add (UI)".

**Plan:**
1. In `handleCreateProject`, after `compose.Parse` succeeds, iterate `parsed.Services` and upsert one service row per compose service (name + image_ref from the parsed spec; digest/container_ids empty until containers are matched).
2. Better (do both): teach the reconciler to match discovered containers to manual projects by compose label, drop the `manualNames` guard for the *service-upsert* pass while keeping it for project-row fields (so discovery never clobbers a manually registered project's config_files/working_dir).
3. Tests: create manual project → service rows exist with image refs; reconcile with matching running containers → digests + container_ids filled; reconcile must not overwrite the manual project row itself.

### A2. Failed apply permanently hides the update

**Where:** `internal/job/worker.go:400` (`failApply` sets `status='failed'`), `internal/store/updates.go:113` (`RecordDrift` preserves status on conflict).

**Problem:** After a failed apply, re-detection of the same digest hits the existing `(service_id, to_digest)` row and preserves `failed`. The update never returns to `available`; `ListOpen` filters it out; the dashboard shows "Up to date" while drift persists. The same digest can never be re-applied until the upstream publishes a *new* digest.

**Plan:**
1. In `RecordDrift`, on the existing-row branch, resurrect `failed` → `available` (only `failed`; `dismissed` and `applied` are user/system intent and stay preserved, `applied` doubly so, it protects the rollback-respect invariant documented in `runRollback`).
2. Test: fail an apply → re-run detection with same digest → update open again; dismissed/applied rows remain untouched by re-detection.
3. (Optional UI follow-up) surface `failed` updates with a "Retry" affordance instead of waiting for the next scan tick.

### A3. Auto-update "Inherit" never inherits

**Where:** `internal/store/projects.go:145` (`EffectiveAutoUpdate`), `web/src/components/settings/AutoUpdateToggles.tsx` (tri-state Select on/off/inherit→true/false/null).

**Problem:** `EffectiveAutoUpdate` requires the service override to be explicitly `true`: `p.AutoUpdateEnabled && s.AutoUpdateEnabled != nil && *s.AutoUpdateEnabled && !s.Pinned`. The UI offers "Inherit" (null) implying the project setting cascades, but null evaluates as OFF. Project-on + service-inherit → never auto-applies. UI semantics contradict the backend.

**Plan:**
1. Change to `p.AutoUpdateEnabled && (s.AutoUpdateEnabled == nil || *s.AutoUpdateEnabled) && !s.Pinned`. Safety posture is preserved: projects default OFF, so nil-inherit still yields OFF globally by default; explicit service `false` still vetoes.
2. Table-driven test over all six (project × service-tristate) combos, plus pinned veto.
3. Update AutoUpdateToggles copy to "Inherit (project setting)".

### A4. UI-edited settings silently require a restart

**Where:** `cmd/dockbrr/main.go:131-209`: `poll_interval_seconds`, `air_gap_mode`, `cache_ttl_seconds`, `concurrency`, `health_timeout_seconds`, `health_poll_seconds` all read once at boot.

**Problem:** The settings screen saves these keys; nothing re-reads them. Worst case is air-gap: the user enables it and network egress continues until restart. A safety-relevant staleness. Also `cache_ttl_seconds` is read at boot but is not even in the settings whitelist (`internal/httpapi/settings.go:16`), so it is boot-configurable only via direct DB edit.

**Plan:**
1. Hot-reload the cheap ones by reading at point-of-use, mirroring the existing `tokenFn` pattern:
   - `airGap` becomes `func() bool` passed to `changelog.NewResolver`.
   - Detector reads `cacheTTL` per `Detect` call (or accepts a `func() time.Duration`).
   - Applier reads health timeout/poll per job.
   - Scheduler loop re-reads the interval each tick and resets its ticker on change.
2. `concurrency` genuinely needs a restart → `GET /api/settings` returns a `restart_required: [...]` list; the settings screen shows a "restart to apply" badge after saving one of those keys.
3. Add `cache_ttl_seconds` to the settings whitelist.
4. Test: flip `air_gap_mode` in the store → next `Resolve` skips network sources without restart.

### A5. Compose runner never passes the project name (`-p`)

**Where:** `internal/compose/runner.go:51` (`buildArgs` emits `-f` + `--project-directory` only).

**Problem:** A stack originally launched with `docker compose -p custom` (project name ≠ directory basename, no `COMPOSE_PROJECT_NAME` in `.env`) → dockbrr's `pull/up` operates on a *different* compose project namespace → duplicate containers started side-by-side with the old stack. The correct name is already known (compose label, stored as `projects.name`).

**Plan:**
1. Add `ProjectName` to `RunSpec`; `buildArgs` emits `-p <name>` when non-empty.
2. Thread `proj.Name` through `PullSpec` / `UpSpec` / the rollback specs in `worker.go` / the `/preview` endpoint.
3. Update preview + spec-builder tests (command strings now include `-p`). Low risk, high correctness value.

### A6. Version delta effectively unimplemented

**Where:** `internal/detect/detect.go:100` (toVer from OCI label only, else `= fromVer`), design §5.4.

**Problem:** Design calls for parsing semver from tags and querying `/tags/list` for newer tags. Reality: `to_version` comes only from the `org.opencontainers.image.version` label; when absent it equals `from_version`, so severity is almost always `digest-only`. Cascades: no severity colors on the dashboard, and the changelog GitHub source keys off this version (`internal/changelog/resolver.go:73`) so it mostly misses for moving tags. Two headline features are hollow.

**Plan:**
1. Add `ListTags(ctx, repo) ([]string, error)` to `registry.Resolver` via go-containerregistry `remote.List` (same anonymous-first + creds-on-401 shape as `get`).
2. In the detector, when the tracked tag parses as semver: list tags, filter parseable semver greater than current, pick the max, resolve that tag's digest as the candidate, set `to_version`/`tag`, severity from the delta. When the tag is moving (`latest` etc.): keep the digest compare; version from label only (current behavior).
3. Cache the tag list alongside `image_remote_state` (new column or a JSON blob) under the same TTL; a rate-limited/failed list is non-fatal (fall back to current digest-only behavior).
4. Also fix the small cache-path wart at `detect.go:66`: a fresh-cache drift records severity `digest-only` unconditionally, after this change, route the cache hit through the same version-classification path.
5. Tests: fake resolver with tag list → major/minor/patch computed; list failure → digest-only fallback; moving-tag repo untouched.

### A7. Registry error / rate-limit state invisible

**Where:** written at `internal/detect/detect.go:77` (`image_remote_state.status` = `error`/`rate_limited`), exposed nowhere. Design §5 requires a per-registry status badge so a rate-limited image is "visible rather than a silent stall".

**Plan:**
1. Add `RemoteStates` to `httpapi.Deps`; extend `serviceDTO` with `check_status` and `last_checked` (from `image_remote_state.resolved_at` for the service's repo:tag). Alternative: a separate `GET /api/registry-status`; the DTO join is simpler and also feeds the dead "Last checked" column (C3).
2. UI: warning icon on affected rows with tooltip ("registry rate-limited, last checked 12m ago"); populate the Last-checked column via the existing `relative()` presenter.
3. Handler test + presenter test.

### A8. Dead services and empty projects linger forever; status badge blind to real states

**Where:** `internal/store/services.go:106` (`MarkGone` sets `state='gone'`), `internal/httpapi/projects.go` (no filter), `web/src/components/StatusBadge.tsx:22` (`computeStatus` knows only `pinned` and the never-occurring `state === "error"`).

**Problem:** A removed container's service stays rendered as a normal row with an "Up to date" badge forever. Discovered projects whose services are all gone keep their header. `error` is not a real Docker state, so `exited`/`dead`/`restarting` containers also show "Up to date".

**Plan:**
1. Extend `computeStatus`: map real states: `exited`/`dead` → "Stopped" (danger), `restarting` → "Restarting" (warning), `gone` → "Gone" (muted).
2. Default-hide `gone` rows behind a "Show removed" filter toggle (keep the data; it feeds history).
3. Project cleanup: hide (or offer explicit "Remove") project rows whose services are all `gone`; avoid silent auto-delete, history hangs off services.
4. Tests: reconcile without a container → row flagged gone and hidden by default; exited container → "Stopped" badge.

### A9. Check button does nothing visible

**Where:** `internal/httpapi/updates.go:105` (202 + goroutine), `web/src/hooks/mutations.ts` (`useCheck` invalidates immediately), no toast.

**Problem:** The invalidation races the detached detection goroutine, so the refetch returns pre-check data. No last-checked column, no toast, no auto-refresh → the button appears inert.

**Plan:**
1. Make `handleCheck` synchronous (detection is a bounded registry GET; keep a server-side timeout), return after `CheckService` completes; the existing `onSuccess` invalidation then observes the result.
2. Toast the outcome via the already-mounted-but-unused sonner `Toaster`: "Checked <svc>: update found / up to date / registry error".
3. Test: msw handler → mutate → assert updates query refetched after response and toast rendered.

### A10. No push freshness anywhere

**Problem:** Scheduler finds an update → UI shows it only on tab refocus. A job reaching terminal state doesn't refresh the dashboard (known gap from the whole-branch review). Design lists SSE as the core stream transport, but the only SSE endpoint is per-job logs.

**Plan:**
1. Stopgap (5 lines, ship immediately): `refetchInterval: 30_000` on `useProjects` + `useUpdates`.
2. Proper: `GET /api/events/stream` (SSE) broadcasting `{type: detected|job_finished|reconciled, service_id?, job_id?}` from detector/applier/reconciler via a small in-process pub-sub; a frontend hook subscribes and maps event type → query invalidation. Reuse the EventSource lifecycle pattern from `useJobLog` (test seam included).

## B. Missing vs design (scope gaps, not bugs)

| Gap | Design ref | Note / micro-plan |
|---|---|---|
| **Update DTO starves the UI** | §8 drawer | `updateDTO` (`internal/httpapi/updates.go:11`) omits `from_version`, `to_version`, `detected_at`, `changelog_text`. Backend caches changelog text; the drawer renders `<Changelog markdown="" />` (`ReviewDrawer.tsx:83`). The markdown path is dead code. **Plan:** add the 4 fields to the DTO + `Update` TS type; pass `update.changelog_text` into `Changelog`. Sanitizers already exist on both sides. ~20 lines; instantly lights up a headline feature. |
| **Service detail unreachable** | §8 | Route `/service/$id` exists; zero links point to it. **Plan:** service-name cell becomes `<Link to="/service/$id">`; breadcrumb/back link on the detail page. |
| Settings export/import | §9 | Absent (was deliberately not pulled into Phase 7). JSON round-trip endpoint pair + settings-screen buttons. |
| `unmanaged` project marking | §7 | Compose file gone is only discovered at apply-time precheck. Plan: reconciler stats config_files; missing → flag project `unmanaged`, badge in UI, disable apply. |
| Docker events subscription | §3 | Reconcile is a 60s timer only. Plan: `client.Events()` subscription triggering a debounced reconcile; timer stays as fallback. |
| Jobs list / history UI | §4 | No `GET /api/jobs`; a running scheduler auto-apply is invisible unless the job id is known. Plan: paged jobs endpoint + a simple Jobs screen (status, type, service, duration, link to log). |
| Post-success prune + disk report | §7 | Absent; design says default-off prune option. Low priority. |
| Project-scope apply closes only the target service's update | §7 | Sibling services updated by the same `up` keep stale rows until the next reconcile/scan. Plan: after project-scope success, re-locate + `UpdateRuntime` every service in the project and close their matching updates. |
| Rate-limit backoff headers | §5 | Detector records `rate_limited` but there is no backoff; the TTL cache is the only throttle. Acceptable v1; note for A6's tag listing (heavier on Hub quotas). |

Explicit non-goals still respected: no notifications, no multi-host, no auto-rollback, matches §1.

## C. UI polish: density and finish

Diagnosis: default 16px root font with `text-sm` (14px) cells and `p-3` (12px) padding reads oversized/airy; the table is full-bleed with no card chrome; no stats row; text-only header; "Loading…" strings; drawer/dialog animation classes are inert (`tailwindcss-animate`/`tw-animate-css` never installed, known T10-M2); no icons; no favicon. A denser, more finished feel needs a smaller type scale, tighter row rhythm, card-wrapped tables, stat tiles, icon buttons, and skeletons.

Each item below is independently shippable.

### C1. Density + type scale (~1 day), biggest visual delta

- `ui/table.tsx`: `TableCell` → `px-3 py-2 text-[13px]`; `TableHead` → `h-9 text-[11px] uppercase tracking-wide` (dense column headers). Digests/mono spans → `text-xs`.
- Row actions → icon buttons (`Eye` for review, `RefreshCw` for check; lucide-react already a dep) with `Tooltip` (primitive exists); rows land ≈36px.
- Wrap the dashboard table in a card: `rounded-lg border border-slate-200 dark:border-slate-800 overflow-hidden`; page container `mx-auto max-w-screen-2xl px-4` in `AppLayout`.
- Sticky header: `TableHeader` gets `sticky top-0 z-10 bg-white dark:bg-slate-950`.
- Install `tw-animate-css` (Tailwind-v4 successor to `tailwindcss-animate`), import once in `index.css` → the existing dialog/drawer `animate-in`/`slide-in` classes start working with zero component changes.

### C2. Stats header (~0.5 day frontend; small backend piece)

- New `DashboardStats` above `Filters`: tiles for **Services**, **Updates available**, **Pinned**, **Stopped/Failing**, **Last scan**.
- First four derive client-side from `useProjects` + `useUpdates`. **Last scan** needs data: add a tiny `GET /api/status` returning `{last_check_all, next_check_due, docker_reachable, jobs_running}`: also answers "is the scheduler alive", currently unobservable. (Depends on A7/A8 for per-service check state; the status endpoint itself is independent.)
- Tiles are clickable → set the corresponding filter.
- Style: shadcn stat card: border, `text-2xl font-semibold` value, `text-xs` muted label. Follow dataviz/stat-tile conventions; consistent in light+dark.

### C3. Row information density (~0.5 day; wants A7 data)

- Name cell: service name links to detail (B item); standalone rows show the project relationship dimmed.
- Current-image cell: `repo:tag` line 1; line 2 = short digest + "checked 12m ago" via the existing `relative()` presenter fed by A7's `last_checked`.
- Status badge upgrades per A8 (Stopped/Restarting/Gone).
- Severity chip becomes meaningful once A6 lands (real major/minor/patch).

### C4. Feedback layer (~0.5 day)

- sonner toasts on: check result (A9), apply enqueued, dismiss, settings saved, and **every mutation `onError`** (closes deferred minor T12-M3, currently a failed Apply/Dismiss leaves the drawer open with zero feedback).
- Skeleton rows (`animate-pulse`) replace the "Loading…" string.
- Empty states: no projects → "No workloads discovered, is the Docker socket mounted?" with a docs pointer; filtered-to-empty → "No services match".

### C5. Header / nav (~0.5 day)

- Logo mark + wordmark left; nav as underlined tabs with an active state (router `activeProps` already wired, just style it).
- Right side: registry-status indicator dot (A7), theme toggle, user dropdown (logout inside) using the installed-but-unused `@radix-ui/react-dropdown-menu`.
- Favicon + proper `<title>` in `web/index.html`.

### C6. Font (optional)

- Add `@fontsource-variable/inter` (bundled by Vite, preserves the no-CDN/CSP invariant), import in `main.tsx`, point `--font-sans` at it.
